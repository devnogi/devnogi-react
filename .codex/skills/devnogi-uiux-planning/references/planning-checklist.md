# Planning Checklist

## Output structure

- Goal
- Primary user action
- Sections and hierarchy
- Reused components and hooks
- New components only if necessary
- Mobile, tablet, and desktop behavior
- Light mode and dark mode behavior
- Required states
- Target files
- Acceptance checks
- Post-build lint reminder

## Guardrails

- Match the existing DevNogi style before proposing novelty.
- Reject plans that ignore dark mode or responsive behavior.
- Prefer incremental changes over large visual resets unless requested.
- Keep the plan concrete enough that another engineer can implement it directly.
